SK에서 근무하고 계신 데이터 엔지니어 분과의 인터뷰를 진행했다. 

네트워킹하면서 느낀건데 이젠 무작정 어떻게 하냐고 질문할 때는 지난 것 같다. 좀 더 구체적인 문제가 생길 때 커피챗을 하는 것이 좋을 것 같다. 
시간이 부족해서 준비한 모든 질문에 대한 답변을 듣진 못했지만, 다행히 중요도 순서로 나열해서 필요한 답변은 다 들을 수 있었다. 

## notes 
1. 관련 회사 도메인이 어느정도 갈 것인가? 경쟁력이 있는 분야를 서치
2. 현재 하고 있는 업무
    1. 통신 데이터 
    2. Raw 데이터 - 데이터 etl, 모니터링 시스템 등 개발 및 관리
    3. 데이터 분석 환경 개발 및 관리 - 텔레콤 인프라 센터 구성원들이 분석할 수 있는 환경 제공 (인프라 데이터큐브)
3. 비전 관련 
    1. 엔지니어링 커리어 패스 - 국내 환경이 그닥 성장에 도움되는 것이 아님
        1. 죽어도 엔지니어 패스가 가고 싶다(관리자 x) - 우리나라가 적합한 환경 아님. 해외로 가야함. 어느 시점되면 매니저가 되어야 하는 것…
            1. 요즘은 기술을 오픈소스가 선두함. 오픈소스를 기두로 좋은 상용 솔루션이 나옴. Apach top level  프로젝트에 기여하는게 쉽지도 않고, 뭐 이런저런. 
            2. 기술에 관심 많으면 해외 취업 적극적으로 
            3. 기술 자체에 목적을 두면 쉽지 않다. 기술은 도구이다. 도구를 잘 사용하는 것이 좋다. 회사 목표를 실현해주는 기술자가 좋은 기술자…(어떤 성과를 내었느냐가 중요하다. 회사가 인정해주는 가치를 전달할 수 있어야함.)
            4. 기술 플랫폼의 변화
                1. 하나의 플랫폼에 대한 이해도가 높으면 다른 플랫폼 이해 (경쟁관계면) 속도가 매우 빨라짐.
                    1. 스파크 대 플링크
                        1. 비슷한 영역의 스트리밍 처리, 배치 처리, 머신러닝 처리 등
                    2. 어쨌든, 데이터 플랫폼 관점에서 메인 스트림 플랫폼을 디테일하게 써보며 경험, 코드 레벨에서 만지다 보면 다른 플랫폼은 이지
                    3. 회사에서 쓰던거나 메인스트림 제품 사용하기!
                    4. Sql 강의에서 하루에서 한시간씩 이주만 하면 필요한 데이터 뭐 80% 사용 가능. 
            5. LLM 가지고 뭐하는지?
                1. 데이터 레이크 - 데이터 활용할 수 있는 도구 만드는 서비스?
                2. 데이터 카탈로그 관리 서비스 등등
                3. 이런 도구들을 주로 만든다. 
                4. 그래서 RAG + 해서 SQL 쿼리 작성해주고 분석 서비스까지 확장
            6. 그래서 이런 서비스 성공이 현재 목표
	
    4. 직무
        1. 개발 말고도 잘하는 영역이 있어야함. (개발은 기본) 
        2. 그리고 회사 가치에 대해 고민하고 안하고 하는 사람들에 대한 차이가 커짐
        3. 엔지니어들이 잘 못하는 것: 자신이 한 것에 대해 잘 전달하는 것을 못함. (보고) 그러나 회사 생활에서 보고가 중요함. (문서 작업 등등) 내가 한 업무에 대해 회사에 딜리버리 잘하는것이 중요함. 회사는 기술 자체에 대해 관심이 없고 회사는 기술ㅇ로 무엇을 했는지가 중요함. 나를 평가하는 것은 엔지니어가 아니라 다른 사람일 확률이 높다. 이들은 기술에 관심 보다는 가치에 대해 더 관심이 많다. 
        4. 판단이 필요함. 내 리소스를 이 업무에 어느정도 투자해야 하는가? 잘 판단해서.
